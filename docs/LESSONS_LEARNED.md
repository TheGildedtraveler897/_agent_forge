### 2026-04-26 - Host Event-Name Drift Is A Silent-Correctness Class; Surface Validation Alone Is Insufficient

- `Date:` 2026-04-26
- `Context:` The Token-Burn Reconnaissance recon (PATHFINDER_LEDGER §2.3) revealed that `_EVENT_ALIASES["gemini"]` in `scripts/omni_factory.py` was wrong. It mapped canonical events to camelCase names (`preToolUse`, `postToolUse`, `sessionStart`, `stop`), but Gemini CLI v0.39 actually expects PascalCase (`BeforeTool`, `AfterTool`, `SessionStart`, `SessionEnd`, `BeforeAgent`, `AfterAgent`, `BeforeModel`, `AfterModel`, `BeforeToolSelection`, `Notification`, `PreCompress`). Every governed project's `.gemini/settings.json` had been carrying a `preToolUse` key that Gemini's hook dispatcher silently ignored — the seeded `telemetry-guardian` hook never fired on Gemini for the entire prior sprint. The triad validator's `hook_surface_for(gemini)` reported pass=true the whole time because it did a substring match against the command path string (`"telemetry-guardian"` or `"guardian.sh"` in `body`), which finds the JSON regardless of whether the event key is meaningful.
- `Lesson:` Two compounding lessons, each load-bearing. **(1) Host-native event names drift between releases.** Claude Code, Codex CLI, and Gemini CLI all evolve their hook event vocabularies independently; what was canonical in 2025 may be renamed in 2026. The omni-factory's `_EVENT_ALIASES` table is now a curated allow-list that must be re-audited against official docs every recon cycle, the same way `host_sandbox_blocked()` markers are. **(2) Surface validation (file shape, JSON parse, substring presence) is necessary but insufficient.** The only correctness gate that catches host-native semantic drift is **live invocation** — fire a real tool call, observe whether the hook actually fired. The `live-hook-prober` skill exists specifically to close this gap, and its `--probe-invocations` integration into the triad validator is the deeper gate.
- `Architectural Decision:` (a) Corrected `_EVENT_ALIASES["gemini"]` to PascalCase per the 2026-04 hooks reference. (b) Tightened `hook_surface_for()` in the triad validator to require both `guardian_present` (substring command-path check) AND `event_key_present` (the per-host expected event key is a top-level key in the rendered hooks payload). New `_EXPECTED_HOOK_EVENT_KEY = {claude: "PreToolUse", codex: "pre_tool_use", gemini: "BeforeTool"}` constant. (c) New `live-hook-prober` skill at `skills/global/live-hook-prober/` with `SKILL.md` (workflow class, all-host targets) + executable `prober.sh` (POSIX). Detects three escalation modes: `sandbox_blocked` (Codex bwrap), `trust_blocked` (Gemini workspace trust), `headless_permission_constraint` (Claude `-p` mode). (d) `validate-triad-runtime.py` gains `--probe-invocations` flag (default OFF). When ON, runs `live_hook_invocation_for(host, project_root)` after surface checks; matrix entries gain `live_hook_pass` + `live_hook_verdict`. (e) Live verification: Gemini probe returned `verdict: pass`, `observed: block`, exit 0 — the C1 fix is real, not just paper.
- `Evidence:` `scripts/omni_factory.py` `_EVENT_ALIASES["gemini"]` block (corrected); `scripts/validate-triad-runtime.py` `_EXPECTED_HOOK_EVENT_KEY` + tightened `hook_surface_for` + new `live_hook_invocation_for` + `--probe-invocations` flag; `skills/global/live-hook-prober/SKILL.md` and `prober.sh`; commit `f2cea42` for the C1 fix; live Gemini probe artifact `runtime/validation/hook-probe/20260426-035313/gemini/`; triad surface artifact `runtime/validation/triad/20260426-035206/`.
- `Promotion Target:` `docs/CONOPS.md` §Validation Gate — promote "every host-native semantic gets a live invocation probe alongside its surface check" once the same pattern lands for memory bridge (B2) and MCP namespace prefixing (A3). The pattern that worked twice (hook lifecycle, memory layer) is now load-bearing; this is the third confirmation.
- `Status:` promoted (curated allow-list pattern is now load-bearing across hooks, memory bridge, MCP prefixing, and prompt-auto-activator; `_EXPECTED_HOOK_EVENT_KEY` + per-record validator check enforces it in code)

### 2026-04-29 - bootstrap-project.sh --existing Unconditionally Overwrites Governance Docs

- `Date:` 2026-04-29
- `Context:` Sprint "RC Polish + Watchdog Activation" Track B2 exercised `bootstrap-project.sh --existing` against a fake project at `~/Projects/bootstrap-test-existing/` to validate the path documented in `docs/HANDOFF.md` § Manual Follow-Up Items. The fake project was seeded with hand-curated `AGENTS.md`, `docs/CONOPS.md`, and `docs/HANDOFF.md` plus a `src/main.py` with a known fingerprint.
- `Lesson:` `bootstrap-project.sh --existing` mode does NOT preserve existing governance content. Every `cat > "${TARGET_ROOT}/<file>" <<EOF ... EOF` block in the script is an unconditional overwrite, not a check-and-skip. Files that get clobbered: `AGENTS.md` (line 142), `CLAUDE.md` (line 171), `GEMINI.md` (line 190), `docs/CONOPS.md` (line 201), `docs/HANDOFF.md` (line 226). Custom non-standard files (e.g. `src/main.py`) are correctly preserved. The `--existing` flag's name suggests safe coexistence with curated content, but the implementation is "scaffold these files in place, replacing whatever was there."
- `Architectural Decision:` Operators must back up `AGENTS.md` / `CLAUDE.md` / `GEMINI.md` / `docs/CONOPS.md` / `docs/HANDOFF.md` before running `--existing` against a real governed project, or use the script only on greenfield trees. A future hardening pass could (a) detect non-stub content via a content marker and skip with warning, or (b) write `AGENTS.md.bootstrap-stub` alongside instead of overwriting. The current behavior is documented here so the lesson survives across sessions.
- `Evidence:` Test artifact md5 deltas captured 2026-04-29: pre-bootstrap `AGENTS.md=88424c22..., docs/CONOPS.md=ec782737..., docs/HANDOFF.md=3b6f7a0b...`; post-bootstrap `AGENTS.md=c09d4472..., docs/CONOPS.md=29336e6d..., docs/HANDOFF.md=e89d5369...`. `src/main.py` md5 unchanged (`100e360c...`). `scripts/bootstrap-project.sh:124-129` confirms `--existing` only requires the target directory to exist; lines 142, 171, 190, 201, 226 confirm unconditional `cat >`. Bootstrap output also showed sync-gemini correctly refused to clobber unmanaged `~/.gemini/GEMINI.md` — that protective pattern exists for global Gemini context but not for project-local governance docs.
- `Promotion Target:` `scripts/bootstrap-project.sh` itself (add content-marker guard); `docs/HANDOFF.md` § Manual Follow-Up Items (add a backup-first warning to the operator instruction).
- `Status:` active

### 2026-04-29 - Append-First Needs A Bounded-Decay Counterpart; Auto-Loaded Ledgers Are A Hidden Tax

- `Date:` 2026-04-29
- `Context:` `docs/LESSONS_LEARNED.md` (46 KB / 188 lines) and `docs/HANDOFF.md` (37 KB / 253 lines) are imported on every Claude and Gemini session start through `CLAUDE.md` and `GEMINI.md`. With `CONOPS.md`, the factory `AGENTS.md`, and the parent-tree `AGENTS.md`, the per-session auto-load load was ~96 KB before any task even began. Most of the bulk was historical wisdom that had already been promoted into doctrine — the rule was already in `AGENTS.md` Rules or `docs/CONOPS.md` or enforced in code, but the original ledger entry kept riding along on every session. Append-first kept the ledger truthful; nothing kept the load cost bounded.
- `Lesson:` Append-first is necessary but not sufficient for an auto-loaded knowledge surface. Once an entry's lesson has been encoded in doctrine or code, the entry's job is done — keeping it in the auto-loaded file pays the load cost twice. The bounded-decay counterpart needs four properties: (1) act only on entries the upstream skill has explicitly marked `Status: promoted`; (2) verify the promotion claim against on-disk evidence before archiving (extract backtick-quoted file paths from the parenthetical and confirm at least one resolves); (3) preserve wisdom verbatim in an on-disk archive — never summarize, never lose; (4) be reversible (restore round-trips an archived entry back to the main file). The same pattern applies to `HANDOFF.md` sprint sections: keep the most recent N detailed, archive older ones to `docs/archive/SPRINTS.md` with a compact summary table.
- `Architectural Decision:` `policies/distillation.json` (schema v1) authorizes per-target retention rules. Two new global skills implement them: `lesson-distiller` (compacts `LESSONS_LEARNED.md`) and `handoff-archiver` (compacts `HANDOFF.md`). Per the one-skill-one-purpose rule reaffirmed in the 2026-04-28 prompt-auto-activator lesson, distillation is a separate skill from `sprint-harvester` — the harvester captures, the distiller compacts. The integrity gate is promotion-claim verification: the entry's `Status:` parenthetical must name a doctrine path that resolves on disk, otherwise the entry is flagged and left in place. `validate-triad-runtime.py` records `distillation_pass` per host (uniform check across all three; distillation is canonical-side, not host-rendered). Cadence binds to RC/milestone events via a new Step 6 in `branch-finisher`. `branch-finisher` Step 6 runs both skills in dry-run, presents diffs, and requires typed `--yes` confirmation before either applies changes. Dry-run measurement: `LESSONS_LEARNED.md` 46.5 KB → ~15 KB (-67%), `HANDOFF.md` 36.8 KB → ~10.6 KB (-71%), combined ~83 KB → ~26 KB (-69%).
- `Evidence:` `policies/distillation.json` v1; `skills/global/lesson-distiller/SKILL.md` and `distiller.py` (subcommands: `dry-run`, `distill --yes`, `verify`, `restore`); `skills/global/handoff-archiver/SKILL.md` and `archiver.py` (subcommands: `dry-run`, `archive --yes`); `tests/test_distiller.py` (13 OK covering entry parsing, promotion-claim verification positive/negative/no-paths, sprint detection, summary-table construction, idempotency, policy shape); `scripts/omni_factory.py` `verify()` (distillation policy validation block, fail_at_bytes WARN signal); `scripts/validate-triad-runtime.py` (`distillation_surface_for(host, project_root)` function, `distillation_pass` matrix entry, `distill+/distill-` per-host status tag); `docs/CONOPS.md` § Knowledge Distillation (Bounded Decay); `AGENTS.md` Rules (append-first paired with bounded-decay rule); `skills/global/sprint-harvester/SKILL.md` Rule 8 (downstream boundary note); `skills/global/branch-finisher/SKILL.md` Step 6 (milestone-distillation hook).
- `Promotion Target:` `docs/CONOPS.md` § Authoring Pattern: Policy + Renderers + Capability — once distillation has been exercised in two real RC integrations and the dry-run output has been operator-approved at both, promote "policy + skills + verifier + triad gate" as the sixth confirmation of the rhythm (alongside hooks 2026-04-24, memory layer 2026-04-25, memory bridge 2026-04-27, MCP namespace 2026-04-27, prompt-auto-activator 2026-04-28).

### 2026-04-30 - Distillation Diffs Are Rebase-Fragile; Verify Must Be Bidirectional

- `Date:` 2026-04-30
- `Context:` First operator-approved distillation pass shipped on `feat/distillation-pipeline-wip` (commit `897babf`): 14 promoted lessons archived, all 14 index pointers landed cleanly, `verify` reported `ok=true`. The branch was then rebased onto `feat/gsd-extraction-foundation` (which had reset to `origin/master` and added three commits: branch-discipline guardrail, managed-state follow-up, GSD assimilation plan). The rebase succeeded with no reported conflicts, all 33 tests passed, the verifier exit 0. Master was fast-forwarded. Only on a manual recount (`grep -c "→.*archived"`) did it surface that only **5 of 14 index pointers** had landed — the other 9 entries were orphaned in the archive, with no way to find them from the auto-loaded main ledger. `verify` had reported `ok=true` because it only checked one direction (pointer → archive entry).
- `Lesson:` Two compounding lessons. **(1) Distillation diffs are uniquely rebase-fragile.** A distillation pass is a structurally large, line-based diff (delete N entries, add N pointers). When git replays such a diff during rebase against a base whose context has shifted, the deletions tend to apply (their context is the entry's body, which is preserved verbatim) but the additions can silently fail (their context is the surrounding lines, which may have been re-flowed by upstream commits). The result: archive grows correctly, main file loses the cross-reference. **(2) Forward verification is insufficient for invertible operations.** Any compaction skill that splits content across two files needs bidirectional integrity: pointer→archive AND archive→pointer. Forward-only check missed the silent corruption entirely; the live `verify` passed and tests passed and the verifier passed — and the file was still broken.
- `Architectural Decision:` Three changes. (a) Extended `distiller.py` `verify_state()` with a reverse direction: every `### YYYY-MM-DD - title` heading in `docs/archive/LESSONS_PROMOTED.md` must have a corresponding `- DATE — TITLE → ... (archived)` pointer in `docs/LESSONS_LEARNED.md`. The result dict gains `archive_entries`, `archive_entries_orphaned` fields; `ok` is gated on both directions being clean. (b) Added `BidirectionalIntegrityTests` in `tests/test_distiller.py` — three tests: shape of the result dict, archive-side `parse_entries` helper still works, live state passes both directions. (c) Operationally: when running distillation on a branch that will later be rebased, prefer to (i) run distillation **after** the rebase, on the rebased pre-state, rather than rebasing a distillation commit; (ii) always run `distiller.py verify` post-rebase as the final integrity gate. Recovery from this incident: reset WIP branch to the pre-distillation rebased state (`33f7ef3`), re-run `distill --yes` on the clean state, force-push, fast-forward master.
- `Evidence:` Recovery commit on `feat/distillation-pipeline-wip` adds the bidirectional check + 3 new tests + this lesson. Forward-only verify on the broken state: `index_pointers=5, archive_entries_missing=0, ok=true` (false positive). Bidirectional verify on the same state would have caught it: `archive_entries=14, archive_entries_orphaned=9, ok=false`. Post-recovery state: 14 pointers, 14 archive entries, 0 orphaned, `ok=true`. Live distillation re-run on the recovered state: `archived: 14 | post: 19447 | delta: 31262` (matches the original test-run measurements byte-for-byte, confirming the recovery is canonically equivalent).
- `Promotion Target:` `AGENTS.md` Rules — after the next sprint that exercises distillation through a rebase, consider promoting "run `verify` post-rebase before fast-forwarding master" as a named rule alongside the existing append-first/bounded-decay rule. Also `skills/global/branch-finisher/SKILL.md` Step 6 — add an explicit pre-merge `distiller.py verify` check.
- `Status:` active
